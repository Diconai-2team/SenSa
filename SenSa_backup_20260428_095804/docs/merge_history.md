# 팀원 작업물 병합 이력

이 문서는 팀원의 작업 브랜치에서 가져온 변경사항을 항목별로 분석하고,
**어떤 부분을 왜 반영했고** / **어떤 부분을 왜 반영하지 않았는지** 를 기록합니다.

향후 팀원과 코드 방향을 조율하거나 제 판단 근거를 되짚을 때 참고하세요.

## ✅ 반영된 변경 (2건)

### B3 — 영향 센서 ID 알람 메시지 노출

**팀원 의도 번호**: ④

**무엇이 개선되는가**

알람 메시지에 원인 센서의 device_id 와 상태가 포함됩니다.

| 구분 | 이전 (내 코드) | 이후 (병합 후) |
|---|---|---|
| 메시지 예 | `"김재승 주의구역 진입 (센서 주의)"` | `"김재승 주의 상태 진입 (sensor_01 caution, sensor_03 caution)"` |
| 상태 섞임 시 | 구분 불가 | `(sensor_01 danger, sensor_03 caution)` |
| 지오펜스 내부 알람 | 기존 포맷 유지 | `"김재승 위험구역 진입 (고온구역 A)"` 그대로 |

**공학적 근거**

- **ISO 45001** (안전보건경영시스템) 요구사항: 사고 조사 시 근본 원인 추적성
- **정보 엔트로피 증가**: 알람 당 식별 정보가 `O(log N_구역)` → `O(log N_센서 + log 3)` 로 확대
- **오탐 추적**: 특정 센서의 장기 caution 발생 시 고장·교정 필요성 식별 가능

**구현 범위**

- `alerts/services.py`:
  - `_build_message()` 시그니처에 `influencing_sensors: list | None = None` 추가
  - `_sensor_suffix()` 내부 헬퍼로 센서 ID 포매팅 로직 캡슐화
  - `evaluate_worker()` 시그니처에도 동일 인자 추가 (하위 호환: 기본값 None)
- `dashboard/views.py`:
  - `_compute_nearby_sensor_status()` 반환을 `str` → `tuple[str, list]` 로 확장

**우선순위 규칙** (중요)

메시지 suffix 결정 순서:
1. `zone_name` 이 있으면 `"(고온구역 A)"` — 지오펜스 내부 알람은 **기존 포맷 유지**
2. 없고 `influencing_sensors` 가 있으면 `"(sensor_01 danger, sensor_03 caution)"`
3. 둘 다 없으면 `"(센서 주의)"` 같은 기본 suffix

→ **지오펜스 기반 알람은 내 기존 표현 그대로, 센서 기반 알람만 구체화됨**.

### B1 — 근접 센서 계산 시 normal 스킵

**팀원 의도 번호**: ③

**무엇이 개선되는가**

거리 계산 전에 `status == 'normal'` 센서는 early continue 로 스킵.

```python
for s in sensors:
    sensor_status = s.get('status', 'normal')
    if sensor_status == 'normal':
        continue  # ← 이 한 줄이 B1 의 본체
    # ... 거리 계산
```

**공학적 근거**

- **복잡도**: `O(N·M)` → `O(N·k)` (N=작업자 수, M=전체 센서 수, k=비정상 센서 수)
- 현재 시연 규모(센서 5개, 대부분 normal)에서 `sqrt` 호출이 최대 5회 → 대부분 1~2회
- **실제 성능 이득은 미미**하지만, **의도 명확화** 측면에서 유의미
  - 읽는 사람이 "왜 모든 센서를 다 계산하지?" 라는 의문을 갖지 않게 됨
- 팀원의 "센서 기준 탐지" 서술은 수학적으로는 동어반복 (`|A-B| == |B-A|` 대칭)
  이지만, 이 스킵 한 줄이 실질적 개선 포인트

**구현 범위**

- `dashboard/views.py` 의 `_compute_nearby_sensor_status()` 내부 루프
  - 함수명은 `_compute_nearby_sensor_status` 그대로 유지 (내 기존 이름)
  - 팀원의 `_get_sensor_influence` 이름은 채택 안 함 (작업자 시점 네이밍이 더 명확)

## ❌ 반영하지 않은 변경 (2건)

### B2 — 지속 알람을 시간 기반 → 틱 기반으로 전환

**팀원 의도 번호**: ② (일부)

**무엇이 바뀌는가 (팀원 버전)**

| 항목 | 내 코드 | 팀원 코드 |
|---|---|---|
| 상수 | `RE_ALARM_INTERVAL_SEC = 60` | `SUSTAINED_ALARM_TICKS = 5` |
| 판정 기준 | `(now - last_alarm_at) >= 60` | `sustained_ticks >= 5` |
| state_store 필드 | `last_alarm_at` (timestamp) | `sustained_ticks` (counter, 추가됨) |

**왜 반영하지 않았는가**

1. **주기 정책이 과공격적**
   - 팀원이 동시에 제안한 B4 (3초 틱) 와 결합하면 **5틱 × 3초 = 15초** 주기 재알림
   - **ISA-18.2** (알람 관리 표준) 에서 알람 피로(alarm fatigue) 방지를 위해
     권장하는 재알림 주기는 통상 **30초 ~ 5분**
   - 15초 간격 반복 알림은 관리자 인지 부하를 가중시켜 **진짜 중요한 새 알람 놓칠 위험**

2. **스케줄러 주기와 강한 커플링**
   - 시간 기반: 틱을 1→2초로 바꿔도 "60초 재알림" 은 그대로
   - 틱 기반: 틱 주기를 바꾸면 재알림 주기도 함께 변동 → 설정 실수 위험

3. **구조적 강건성**
   - 서버 재시작 후: 시간 기반은 `last_alarm_at` 이 Redis 에 남아 즉시 복구
   - 틱 기반은 카운터 리셋되어 매번 5틱 대기

4. **수정 영역이 큼**
   - state_store 스키마 확장 (sustained_ticks 필드 추가) 필요
   - evaluate_worker 의 지속 판정 로직 전체 재작성
   - "작은 수정" 기준을 명확히 초과

**트레이드오프 인정**

- 팀원 방식의 **장점**: 테스트 용이성 (시간 mock 불필요), 시스템 시계 변경에 견고
- 그러나 산업안전 관제의 **핵심 요구(과도한 알림 억제)** 와 상충하여 제외

### B4 — FastAPI 스케줄러 틱 주기 1초 → 3초

**팀원 의도 번호**: ② (일부)

**왜 반영하지 않았는가**

1. **실시간성 저하**
   - 관제 시스템 업계 표준 틱 주기: 1~2초 (SCADA, HMI, 실시간 DCS)
   - 3초는 용인 범위지만 "관제" 라는 제품 성격에 다소 느림
   - 지도 위 작업자 마커가 3초마다 뚝뚝 끊기게 보여 사용자 경험 저하

2. **DB 부하 절감은 현 시연 규모에선 무의미**
   - 1초 기준 분당 420건 INSERT → 3초 기준 140건
   - 현재 SQLite + 센서 5개 + 작업자 2~3명 환경에선 부하 차이 체감 불가

3. **위험 감지 지연 증가**
   - Worst case: 위험 상황 발생 후 감지까지 1초 → 3초
   - 산업 안전 맥락에선 실시간성이 곧 안전 가치

4. **B2 와 커플링**
   - 팀원 구조에선 틱 주기 바꾸면 지속 알람 주기도 함께 변동

**결정**

현행 1초 틱 유지.

## 🔸 보류된 변경 (1건)

### A1 — 지오펜스 존타입/리스크레벨 UI 체계

**팀원 의도 번호**: ①

**무엇이 바뀌는가**

| 항목 | 내 코드 | 팀원 코드 |
|---|---|---|
| ZONE_TYPE_CHOICES | `danger / caution / restricted` | `open / monitored / hazardous / restricted` |
| risk_level 필드 | 있음 (`low/medium/high/critical`) | 제거됨 |
| GeoFence 판정 | `_classify_state` 에 지오펜스+센서 혼합 | 센서만. 지오펜스는 별도 `check_geofence_transitions` |

**왜 반영하지 않았는가**

1. **모델 구조 자체의 변경** — migration 영향, 기존 알람 데이터 해석 재정의 필요
2. **내 알람 아키텍처와 철학적 차이**
   - 내 방식: 작업자 상태 = (지오펜스 + 센서) 종합 판정, critical 레벨 포함
   - 팀원 방식: 작업자 상태 = 센서만, 지오펜스는 진입/이탈 이벤트로 분리
3. **사용자의 명시적 요청**: "기존 변경은 모두 배제"
4. **팀 간 논의 필요** — 단독 병합으로 결정할 사안이 아님

**향후 조율 방향 (제안)**

- 지오펜스 zone_type 체계는 팀 미팅에서 **단일 기준 수립** 후 일괄 반영
- 내 `critical` 상태와 팀원의 `restricted` 구역 처리는 개념 충돌 가능 → 명시적 합의 필요
- 알람 아키텍처(A2) 와 연계된 결정이라 **기능 전체를 한 스프린트에 통째로 리팩터링** 하는 것이 안전

## ✅ 추가 반영 — CSS 일관성 수정 (3건)

### 배경

팀원 코드 분석 과정에서 **CSS 쪽 누락**을 뒤늦게 발견.
이는 팀원의 "새로운 개선" 이 아니라 **내 코드가 원래 갖고 있어야 할 것이 빠져 있던 부분**.

### 문제 진단

내 `alerts/services.py` 는 회복/복귀 전이 시 `alarm_level='info'` 알람을 생성하지만 (9곳),
대시보드 CSS 에서는 **info 레벨용 스타일이 누락**돼 있었음:

| 레이어 | info 처리 | 상태 (변경 전) |
|---|---|---|
| 알람 생성 (`services.py`) | 9곳에서 info 생성 | OK |
| 알람 상세 페이지 (`alarm_list.html`) | `level-badge.info` 있음 | OK |
| 대시보드 alarm-item | `.alarm-item.level-info` (하드코딩 `#3498db`) | 부분 OK |
| 대시보드 토스트 | **`.toast-info` 없음** | ❌ 누락 |
| 대시보드 경고 배너 | **`#alert-banner.level-info` 없음** | ❌ 누락 |
| `--color-info` 토큰 | **없음** | ❌ 누락 |

### 반영한 수정

1. **`base.css`**: `--color-info: #3498db` 토큰 신규 추가
2. **`section_09_map.css`**: `.toast-info` 스타일 추가
3. **`section_10_events.css`**:
   - `#alert-banner.level-info` 스타일 추가
   - 기존 하드코딩 `.alarm-item.level-info { border-left-color: #3498db }` → `var(--color-info)` 로 토큰화

### 공학적 근거

- **일관성 원칙**: 서버가 생성하는 알람 레벨과 UI 가시화 레이어가 일치해야 함
- **Single Source of Truth**: 색상값을 `:root` 에 집중시켜 향후 팔레트 변경 시 한 곳만 수정
- **누락된 시각 피드백 복원**: 회복 알람(info) 이 현재는 기본 스타일(빨간색) 로 잘못 표시될 가능성이 있었음

### 의도적으로 남겨둔 하드코딩

- `section_10_events.css` 의 `.detail-link { color: #58a6ff }`
  - info 토큰(#3498db) 과 다른 **"GitHub-style 링크 파랑"** 의미로 구분
  - 팀원 코드에서도 동일하게 하드코딩 유지됨 → 의미가 다른 용도이므로 별도 토큰화 보류

## 🧪 검증 체크리스트

병합 적용 후 다음을 확인:

### Python 로직 측면

1. 기존 지오펜스 기반 알람 메시지가 **예전과 동일하게** 나오는지
   - `"김재승 위험구역 진입 (고온구역 A)"` 포맷
2. 센서 단독 알람이 발생할 때 **구체 센서 ID가 포함**되는지
   - `"김재승 주의 상태 진입 (sensor_01 caution)"` 형태
3. 여러 센서가 동시에 비정상일 때 **쉼표로 나열**되는지
4. 상태가 섞여 있을 때 **각 상태별로 구분**되는지
   - `(sensor_01 danger, sensor_03 caution)`
5. `sustained` (지속 알람) 재알림이 여전히 **60초 주기**로 나오는지 (B2 미반영 확인)

### CSS 시각 측면

6. 작업자가 `danger→caution` 으로 회복할 때 토스트가 **파란색 info 색조**로 표시되는지
   - 이전에는 `.toast-info` 가 없어서 기본 스타일로 나왔을 것
7. 대시보드 ⑩ 이벤트 패널의 회복 알람이 **파란 왼쪽 테두리**로 구분되는지
8. 알람 상세 페이지(`/dashboard/alarms/`)에서 info 알람 뱃지가 **파란 테두리 + 파란 글자**로 표시되는지
